# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

NWQS is an R package implementing non-linear weighted quantile sum regression for correlated mixture exposures. The package extends classical WQS by:

- quantile-transforming exposures,
- expanding each component into a natural cubic spline basis,
- learning component weights with out-of-bag permutation importance,
- estimating the final mixture effect through repeated holdout validation,
- and using an external bootstrap for valid inference.

The package supports `gaussian`, `binomial`, `poisson`, `quasipoisson`, and `negbin` families. (Conditional logistic regression was supported in 0.1.x; it was removed in 0.2.0 and is planned to return as a dedicated `nwqs_clogit()` function in a later release. Ordinal regression via `MASS::polr` is also planned for a later release.)

## Working principles (v0.2.0 refactor in progress)

This package is being refactored toward a 0.2.0 release that supports a companion applied-domain paper. While that refactor is in progress, **all changes under `R/` must follow the rules below**. AGENTS.md and CLAUDE.md hold an identical copy of this section; they must stay in sync.

1. **TDD.** Any change to a function in `R/` requires a failing test first. Tests live in `tests/testthat/`.
2. **Three core invariants must not break.** Every refactor must preserve:
   - Training/validation split separation: spline shapes and component weights are learned on the training split; the final regression is fit on the validation split.
   - Global spline knot alignment: train, validation, and bootstrap data share one set of `knots`/`Boundary.knots` derived from the same quantile/percentile grid.
   - `rh > 1` is not valid inference. Standard errors reported by `nwqs()` reflect algorithmic (data-splitting) variance only. `nwqs_boot()` is the only path for sampling-variance inference. Do not blur this distinction in code, docs, print methods, or new S3 methods like `vcov()` / `confint()`.
3. **Statistical rigor.** Quantile/percentile transforms, OOB loss functions, weight normalization, SNR definitions, and spline knot strategies each need an explicit mathematical definition in their roxygen documentation. Tests must cover boundaries (NA columns, ties, single-value columns, tiny samples, predict newdata outside training range).
4. **Do not hand-edit `man/` or `NAMESPACE`.** Edit roxygen comments and run `devtools::document()`.
5. **AGENTS.md and CLAUDE.md must stay byte-identical.** Whenever this section (or any other shared content) is changed, update both files in the same commit. `diff CLAUDE.md AGENTS.md` should be empty.
6. **Default values have a single source of truth.** All user-facing default values live in the `NWQS_DEFAULTS` list in `R/zzz-defaults.R`. The `formals` of `nwqs()`, `nwqs_boot()`, and `nwqs_control()` read from `NWQS_DEFAULTS` rather than hard-coding literals. Changing a default means changing one entry.

## Code structure

### Core modeling pipeline

- `R/nwqs.R`: primary implementation of `nwqs()` and `nwqs_boot()`.
  - `nwqs()` performs quantile transformation, creates globally fixed spline knots, runs repeated holdout splits, learns shapes/weights on training data, constructs the final `nwqs` index on validation data, and fits the final GLM.
  - `nwqs_boot()` is the outer bootstrap wrapper used for valid sampling-variance inference.
- `R/weights.R`: internal scoring engines.
  - `permutation_scorer()` fits a single in-bag model and derives component weights from grouped OOB permutation loss increases.
  - `run_oob_permutation()` prepares the spline-expanded design matrix and orchestrates repeated internal scoring calls.
- `R/parallel.R`: `future` plan setup via `configure_parallel_plan()`. This helper is intentionally non-invasive and returns early if the caller already has a non-sequential future plan active.
- `R/utils.R`: shared utilities for quantile transforms, spline expansion, effect extraction, and contrast calculations.

### User-facing methods and plotting

- `R/methods.R`: `print`, `summary`, `plot`, and `coef` methods for `nwqs` and `nwqs_boot` objects.
- Plotting logic is tightly coupled to the structure of fitted objects (`final_weights`, `mean_shapes`, `rh_*`, spline knots/boundaries). If you change fit object fields in `R/nwqs.R`, check plotting and summary methods here as well.

### Simulation and benchmarking

- `R/simulate_data.R`: synthetic data generators for continuous, binary, and count outcomes, plus covariance and covariate generators.
- `R/monte_carlo.R`: helpers for simulation benchmarking, coverage checks, weight-recovery metrics, and benchmark visualizations.

## Architecture details that matter

### The fitting algorithm is a staged pipeline

The package is built around this flow:

1. Transform mixture variables to quantile bins with `trans_quantile()`.
2. Create a spline basis for each component with `wqs_nonlinear_expand()`.
3. On each repeated-holdout training split, estimate:
   - spline shape coefficients per component,
   - component weights from OOB permutation importance.
4. Normalize component shapes to unit variance before combining them with weights.
5. Build a one-degree-of-freedom `nwqs` index and fit the final regression on the validation split.
6. Average coefficients, weights, and shapes across repeated holdout iterations.

When editing this pipeline, preserve the separation between:

- representation learning on the training split, and
- final effect estimation on the validation split.

### Global spline alignment is a core invariant

Spline knots and boundary knots are computed once from the quantile grid and then reused across train/validation/bootstrap data. This prevents basis drift between resamples.

If you modify spline construction, keep the train/valid/bootstrap basis aligned; `wqs_nonlinear_expand()` assumes knots and boundaries are supplied explicitly.

### `rh > 1` is not valid inference

A key modeling assumption throughout the repo is that repeated-holdout variability is algorithmic variance only. The warnings in `nwqs()` and the README are intentional:

- `nwqs()` provides averaged coefficients and RH-based variability,
- `nwqs_boot()` is the intended path for valid confidence intervals and inference.

Do not “simplify away” that distinction when changing summaries, print methods, or docs.
